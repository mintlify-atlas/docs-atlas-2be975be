---
title: "Consuming and Processing Messages from Broccoli"
sidebarTitle: "Consuming"
description: "Consume messages individually or in batches, process them with concurrent workers, and handle ack/reject lifecycles using Broccoli's consume API."
---

Broccoli gives you three consumption patterns, each suitable for a different use case.  All patterns share the same low‑level broker methods but offer a different level of abstraction.

### 1. Blocking consume

`consume(topic, options)` and `try_consume(topic, options)` are the raw API calls.  `consume` will block until a message is available, while `try_consume` returns immediately with `None` if there is nothing to consume.  The returned `BrokerMessage<T>` contains the payload and a `message_id` that you can use to **acknowledge**, **reject**, or **cancel** the message.

<CodeGroup>
```rust
let msg = queue
    .consume::<JobPayload>("jobs", None)
    .await?;
// Process
if process_job(msg.payload).await.is_ok() {
    queue.acknowledge("jobs", msg).await?;
} else {
    queue.reject("jobs", msg).await?;
}
```
</CodeGroup>

### 2. Batch consume

`consume_batch` and `try_consume_batch` let you pull multiple messages in one round.  They are useful when you want to pre‑fetch a set of tasks and then hand them off to a worker pool.

<CodeGroup>
```rust
let batch = queue
    .consume_batch::<JobPayload>("jobs", 10, time::Duration::seconds(5), None)
    .await?;
for msg in batch {
    process_job(msg.payload).await.ok();
    // Ack or reject in a loop
}
```
</CodeGroup>

### 3. High‑level message processor

The convenience methods `process_messages` and `process_messages_with_handlers` spin up a pool of Tokio tasks that continuously pull messages, run a user‑supplied handler, and automatically ack or reject based on the handler's result.

<CodeGroup>
```rust
queue
    .process_messages("jobs", Some(4), None, |msg| async move {
        process_job(msg.payload).await
    })
    .await?;
```
</CodeGroup>

When you need separate callbacks for success and error you can use `process_messages_with_handlers`:

<CodeGroup>
```rust
queue
    .process_messages_with_handlers(
        "jobs",
        Some(5),
        None,
        |msg| async move { process_job(msg.payload).await },
        |msg, _| async { success_handler(msg.payload).await },
        |msg, err| async { error_handler(msg.payload, err).await },
    )
    .await?;
```
</CodeGroup>

#### ConsumeOptions

All consume‑style APIs accept an optional `ConsumeOptions`.  The builder pattern is convenient:

<CodeGroup>
```rust
let options = ConsumeOptions::builder()
    .auto_ack(false)          // default – you must ack/reject manually
    .fairness(true)           // only for Redis
    .consume_wait(time::Duration::milliseconds(10))
    .handler_ack(true)        // whether the handler should automatically ack
    .build();
```
</CodeGroup>

- **`auto_ack`** – if `true`, the message is acknowledged as soon as it is consumed.
- **`fairness`** – enables round‑robin consumption from a fairness queue (Redis only).
- **`consume_wait`** – sleep duration between tight consumer loops; useful to allow Tokio tasks to be cancelled.
- **`handler_ack`** – set to `false` if you want to defer the ack to an external actor.

<Tip>
Set a small `consume_wait` (e.g. 10 ms) to keep the consumer responsive while still giving Tokio the chance to abort the spawned task.
</Tip>

#### Manual lifecycle

The broker exposes `acknowledge`, `reject`, and `cancel` methods.  The `cancel` method removes a message entirely from the queue.

<CodeGroup>
```rust
let msg = queue.consume::<JobPayload>("jobs", None).await?;
// If the message is malformed – drop it permanently
queue.cancel("jobs", msg.message_id).await?;
```
</CodeGroup>
