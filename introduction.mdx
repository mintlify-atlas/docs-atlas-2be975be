Broccoli is a robust, asynchronous message queue library for Rust applications. If you've used Celery in Python and want the same ergonomics with Rust's performance and compile-time type safety, Broccoli provides a familiar API backed by multiple message broker implementations.

Published on crates.io as [`broccoli_queue`](https://crates.io/crates/broccoli_queue), it builds on Tokio for async I/O and uses serde for type-safe message serialization. A single `BroccoliQueue` instance handles publishing, consuming, retries, and scheduling — regardless of which broker you're connected to.

## Key Features

- **Async message processing** — built on Tokio; non-blocking publish and consume operations
- **Multi-broker support** — Redis (default), RabbitMQ, and SurrealDB with a consistent API
- **Configurable retry strategies** — automatic retries with configurable attempt limits
- **Connection pooling** — bb8 for Redis, deadpool for RabbitMQ
- **Type-safe messages** — any `serde`-compatible type as a message payload
- **Message scheduling** — delay messages or schedule them for a specific time
- **Priority queuing** — five priority levels (1 highest to 5 lowest)
- **Fairness queuing** — round-robin consumption across tenant/user groups via disambiguators
- **Concurrent processing** — configurable worker pool via `process_messages`
- **Management API** — optional feature for queue introspection at runtime

## Feature Flags

Broccoli uses Cargo feature flags to keep the dependency footprint small. Enable only the broker you need:

| Feature | Default | Description |
|---------|:-------:|-------------|
| `redis` | ✓ | Redis broker via `bb8-redis` |
| `rabbitmq` |  | RabbitMQ via `lapin` + `deadpool-lapin` |
| `surrealdb` |  | SurrealDB broker |
| `management` |  | Queue introspection API (`queue_status`) |

At least one broker feature must be enabled, or the crate will fail to compile.
